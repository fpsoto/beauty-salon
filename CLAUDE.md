# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project state

Beauty Salon is a .NET MAUI app for a single-user beauty salon administrator (clients, weekly agenda, service catalog, charges, reports). It follows a staged build-out defined by the user:

1. **Fase 1 (done)** — Clean Architecture skeleton + domain model.
2. **Fase 2 (done)** — Persistence (EF Core/SQLite), migrations, audit/soft-delete interceptor, seeder.
3. **Fase 3 (done)** — Application services/use cases, DTOs, validators, and the core ViewModels (Login, Agenda, Client list/detail).
4. **Fase 4 (done)** — MAUI UI: XAML pages for every screen, Shell navigation, remaining CRUD ViewModels (Catalog, Payment Methods) built alongside their pages.
5. **Fase 5 (done)** — Reports, local notifications, settings, i18n (`.resx`, es/en). See the dedicated section below.
6. **Fase 6 (in progress)** — Testing started (see the dedicated section below); optimization/docs not yet addressed.

**The app has been run on-device** (the user's own Android emulator via Visual Studio) and confirmed working end-to-end through Fase 4; Fase 5 additions (Settings/Reports/notifications) have only been verified via `dotnet build` in this dev environment — no Android emulator/device is available here. Budget a device pass over Fase 5 screens before assuming they render correctly.

All C# identifiers (classes, properties, parameters, locals) are English, even though the domain/spec is Spanish — e.g. `Client.Name`/`LastName`, `Appointment.Date`/`StartTime`/`Status`, `AppointmentStatus.Booked`/`Completed`. Only genuine Spanish-language *content* the app shows the user stays in Spanish: seeded display strings (category/service/payment-method names like "Corte Hombre", "Efectivo"), and validation/business error messages returned in `Error.Message` (these are Application-layer strings, not resx-backed — see the i18n note below). Don't reintroduce Spanish property names when extending entities/DTOs.

Git is initialized with a GitHub remote (`origin` → `fpsoto/beauty-salon`, no LICENSE — all rights reserved). One test project exists (`BeautySalon.Application.Tests`) — see the Fase 6 section below.

Target platforms: **Android only today; iOS is kept for a future release** (buildable only from a macOS host — the condition in the `.csproj` drops it on Linux). Windows and MacCatalyst were deliberately removed — do not re-add them without being asked.

## Commands

Requires the .NET 10 SDK with the MAUI workload (`dotnet workload install maui`) and `dotnet-ef` (`dotnet tool install --global dotnet-ef`).

```
dotnet build "Beauty Salon.slnx"                       # build everything (all class libs + MAUI head)
dotnet build "Beauty Salon.csproj" -f net10.0-android   # build the app for Android
dotnet build "Beauty Salon.csproj" -f net10.0-ios       # build for iOS (macOS host only)

# EF Core migrations (run from repo root; the design-time factory needs no MAUI host):
dotnet ef migrations add <Name> --project "BeautySalon.Persistence/BeautySalon.Persistence.csproj" --output-dir Migrations
dotnet ef database update --project "BeautySalon.Persistence/BeautySalon.Persistence.csproj"

dotnet test "BeautySalon.Application.Tests/BeautySalon.Application.Tests.csproj"  # run the test suite
```

**OneDrive caveat**: this repo lives under a synced OneDrive folder. Android/iOS builds occasionally fail with `Access to the path '...' is denied` while MSBuild tries to clear an `obj/Debug/<tfm>/...` intermediate directory — that's OneDrive transiently locking a file, not a real build error. Fix: delete the specific `obj/Debug/<tfm>` folder and rebuild.

Seeded login: **admin / admin123** (single admin user; the schema is already multi-user-ready via `ProfessionalId` FKs, but there's no login UI yet).

## Architecture

Clean Architecture across 5 projects (flat layout, MAUI head unmoved at repo root):

```
BeautySalon.Domain/          <- no NuGet packages, no project references. Pure C#.
BeautySalon.Application/     <- Domain only. Contracts (interfaces), no implementations yet.
BeautySalon.Persistence/     <- Application + Domain. EF Core/SQLite implementation.
BeautySalon.Infrastructure/  <- Application + Domain. Non-MAUI platform services (hashing, clock, current user).
Beauty Salon.csproj          <- Application + Infrastructure + Persistence. MAUI head / composition root.
```

`Beauty Salon.csproj` has an explicit `<Compile Remove>` for the 4 sibling project folders — without it, the MAUI project's default recursive glob would also compile their `.cs` files directly, duplicating every type. Keep that in mind if new sibling class libraries are added.

**Domain** (`BeautySalon.Domain`): `Common/BaseEntity` (Guid v7 PK) → `AuditableEntity` (Created/Updated/Deleted At+By, `IsDeleted`, `Version` as a portable int concurrency token) is the base for every entity, without exception — even simple catalogs like `PaymentMethod`. `Common/Result`+`Error` is the Result-pattern used for *expected* business failures (schedule conflicts, duplicate RUT); real exceptions are reserved for genuinely unexpected IO/DB failures. `Common/IScheduleOccupant` (`Date`/`StartTime`/`EndTime`/`ProfessionalId`) is implemented by both `Appointment` and `ScheduleBlock` so overlap/availability logic can treat them as one merged list instead of duplicating the check per type. `ValueObjects/Rut` validates the Chilean check digit (the one identifier kept as-is — it's a proper-noun acronym, not a translatable word). Entities: `User`, `Client`, `ServiceCategory`, `SalonService` (named to avoid colliding with `...AppService`), `Appointment` (`Date`/`StartTime`/`EndTime`/`Status`/`SuggestedPrice`/`ChargedPrice`/`Discount`/`Tip`/`InternalNotes`), `AppointmentServiceItem` (bridge table with a *snapshot* of price/name/duration at booking time — catalog changes must never alter historical appointments), `ScheduleBlock` (`Type`/`Reason`), `WorkingHours`, `PaymentMethod`, `AppSettings`+`NotificationRule`, `Product` (minimal, no inventory yet). `AppointmentStatus` = `Booked`/`Confirmed`/`InProgress`/`Completed`/`Cancelled`/`NoShow`/`Rescheduled`; `ScheduleBlockType` = `Lunch`/`Vacation`/`Meeting`/`DayOff`/`Unavailable`/`Other`.

**Application** (`BeautySalon.Application`): `Common/Interfaces` holds every port this app depends on — `IRepository<T>` (generic CRUD) + one specific repository interface per aggregate, `IUnitOfWork` (aggregates all repos + `SaveChangesAsync`), `IDateTimeProvider`, `ICurrentUserContext` (read-only: `UserId`/`Username`), `ISessionService` (write-side companion: `SignIn`/`SignOut`, implemented by the same class as `ICurrentUserContext` — see Infrastructure below), `IPasswordHasher`, `IWorkingHoursProvider`, `IScheduleAvailabilityChecker`. `Common/Exceptions` (`ConcurrencyConflictException`, `DataAccessException`) are what Persistence throws instead of ever letting an EF Core/Sqlite exception type cross into Application. `Common/WellKnownIds.AdminUserId` is the seeded admin's fixed id — used by the seeder and as every session-aware ViewModel's defensive fallback (`_currentUserContext.UserId ?? WellKnownIds.AdminUserId`), not as the primary source of identity anymore.

`Features/<Auth|Clients|Catalog|Payments|Schedule>` hold the Fase 3 AppServices, one per aggregate, each with: DTOs, `Create`/`Update` request records, a `FluentValidation` validator per request (validates shape/format only — see below), a `*MappingExtensions` static class (`ToDto()`), and an `I*AppService`/`*AppService` pair. Every AppService method returns `Result`/`Result<TDto>`, injects `IUnitOfWork` only, and follows the same shape: validate → load/check business rules → mutate → `SaveChangesAsync` → map to DTO. `Features/Schedule` additionally holds `WorkingHoursProvider`/`ScheduleAvailabilityChecker` (the concrete implementations of the Fase 1 contracts) and `WeekHelper` (Monday–Sunday week bounds). Business-rule checks that need data access (schedule overlap, duplicate RUT, "category still has services") live in the AppService or `ScheduleAvailabilityChecker`, never in the FluentValidation validators. `ApplicationServiceCollectionExtensions.AddApplication()` registers everything, including `AddValidatorsFromAssembly`.

Note on `AppointmentAppService`: `RescheduleAsync` does **not** edit the appointment in place — it creates a new `Appointment` (status `Booked`) linked via `RescheduledFromAppointmentId`, and flips the original to `Rescheduled`. `CancelAsync`/`ConfirmAsync`/`StartAsync`/`MarkNoShowAsync` all funnel through a private `TransitionAsync` helper that checks the allowed source statuses before mutating — extend that helper (with its optional `mutate` callback) rather than hand-rolling a new transition method.

**Persistence** (`BeautySalon.Persistence`): `BeautySalonDbContext` applies every `IEntityTypeConfiguration<T>` from `Configurations/` via assembly scanning, then in `OnModelCreating` applies the `Version` concurrency token *and* the `!IsDeleted` query filter to every `AuditableEntity` type generically via reflection — don't add per-entity soft-delete filters, extend that loop instead. `Interceptors/AuditableEntitySaveChangesInterceptor` stamps Created/Updated/Deleted(At+By) and converts a `Remove` into a soft-delete (`IsDeleted=true` + `EntityState.Modified`) — repositories call `DbSet.Remove(...)` normally and never need to know this. `Repositories/EfRepository<T>` is the generic base; specific repos add only real query methods (search, date ranges, overlap checks). **`ClientRepository.SearchAsync` filters in-memory, not via SQL** — `Client.Rut` is a `HasConversion` value object, and translating a `Like`/equality check against it via `EF.Property<string>` throws `InvalidCastException` at runtime (a real bug hit and fixed during Fase 3); loading all clients and filtering client-side is deliberate and fine at this app's scale, don't try to push it back into SQL without re-verifying that translation actually works. `UnitOfWork.SaveChangesAsync` is the *only* place that translates `DbUpdateConcurrencyException`/`DbUpdateException` into the Application-level exceptions — this is deliberate, don't add try/catch-and-rethrow elsewhere. `BeautySalonDbContextFactory` (`IDesignTimeDbContextFactory`) lets `dotnet ef` run without the MAUI host (pass `--startup-project` pointing at Persistence itself, or `dotnet-ef` may pick the MAUI head and fail on its multi-targeting). `Seed/DatabaseSeeder` creates the admin user (fixed id `WellKnownIds.AdminUserId`), a starter catalog, payment methods, Mon–Fri working hours, and `AppSettings`+`NotificationRule`s — `PersistenceServiceCollectionExtensions.InitializeDatabaseAsync` runs `PRAGMA journal_mode=WAL`, `Database.MigrateAsync`, then the seeder; called once from `MauiProgram.CreateMauiApp()`.

**Infrastructure** (`BeautySalon.Infrastructure`): `Services/DateTimeProvider`, `BCryptPasswordHasher`. `CurrentUserContext` implements **both** `ICurrentUserContext` and `ISessionService` — it's a real (if simple) in-memory session: `SignIn(userId, username)`/`SignOut()` mutate private fields, `UserId`/`Username` read them back. Registered as a single `Singleton` instance mapped to both interfaces (`services.AddSingleton<CurrentUserContext>(); services.AddSingleton<ICurrentUserContext>(sp => sp.GetRequiredService<CurrentUserContext>()); services.AddSingleton<ISessionService>(sp => sp.GetRequiredService<CurrentUserContext>());`) so both sides see the same session state. There is **no cross-restart persistence** (no `SecureStorage`/`Preferences`) — by design, every app launch starts at `LoginPage` again; `SignIn` only lasts for the current process lifetime. WhatsApp deep links and local notifications live in the MAUI head instead (`Services/`, see the Fase 5 section) — they depend on MAUI Essentials/`Plugin.LocalNotification`, which Infrastructure deliberately doesn't reference.

**Presentation / composition root** (`Beauty Salon.csproj`, `MauiProgram.cs`, `ViewModels/`): `MauiProgram.cs` calls `SQLitePCL.Batteries_V2.Init()` first, resolves the SQLite path via `FileSystem.AppDataDirectory`, wires `AddPersistence(connectionString)` + `AddInfrastructure()` + `AddApplication()`, registers the Fase 3 ViewModels as transient, then runs `InitializeDatabaseAsync()` synchronously (`GetAwaiter().GetResult()`) before returning the built `MauiApp` — migrations/seed must be done before any page can query the database. `RootNamespace` stays `Beauty_Salon` (underscore) even though new projects use `BeautySalon.*` (no underscore) — don't "fix" this, it'd break existing XAML `x:Class` references; the `ViewModels/` folder namespace follows the same `Beauty_Salon.ViewModels` convention.

`ViewModels/ViewModelBase` (`ObservableObject`) exposes `IsBusy`/`ErrorMessage` and a `SafeExecuteAsync(Func<Task>)` wrapper that catches unexpected exceptions, logs via injected `ILogger`, and sets `ErrorMessage` — expected business failures should already arrive as a failed `Result` (use `SetError(result.Error)`), not an exception. **`SafeExecuteAsync` is not reentrant**: never call another `SafeExecuteAsync`-wrapped method from inside one (it no-ops because `IsBusy` is already true) — every ViewModel that needs to refresh itself after a mutation splits a private `...CoreAsync` method out for that (see `AgendaViewModel.LoadWeekCoreAsync`, `ClientListViewModel.SearchCoreAsync`, `CatalogViewModel.LoadCoreAsync`) and calls the core method internally instead.

**Real session/login** (`ViewModels/LoginViewModel.cs`, `Services/ISessionService`+`CurrentUserContext`): `LoginViewModel.LoginAsync` calls `IAuthAppService.LoginAsync` (validates credentials, returns `LoginResultDto`) then `ISessionService.SignIn(result.Value.UserId, result.Value.Username)` on success — this is what actually populates `ICurrentUserContext` for the rest of the app run. The four ViewModels that need "the current professional's id" (`AgendaViewModel`, `AppointmentFormViewModel`, `ReportsViewModel`, `SettingsViewModel`) each expose a `private Guid ProfessionalId => _currentUserContext.UserId ?? WellKnownIds.AdminUserId;` computed property instead of caching a field in the constructor — computed, not cached, in case session state ever changes within a still-alive ViewModel instance; the `?? WellKnownIds.AdminUserId` fallback is defensive only, since these pages are unreachable pre-login. `SettingsPage`'s "Cerrar sesión"/"Log out" button (`OnLogoutClicked`) calls `ISessionService.SignOut()` + `IPersistedSessionStore.Clear()` then swaps the window to a fresh DI-resolved `LoginPage`, mirroring the same `Application.Current.Windows[0].Page = ...` swap pattern `LoginPage.xaml.cs` already uses for the opposite direction.

**Persisted session across restarts** (`Services/IPersistedSessionStore`/`PersistedSessionStore`): survives app restarts for up to `App.MaxSessionAge` (30 days, a UX choice for a single-owner device, not a security boundary) via plain MAUI `Preferences` (not `SecureStorage` — only a user id/username is stored, no credential/token, so `SecureStorage`'s keystore complexity and occasional cross-OS-upgrade breakage aren't worth it). `LoginViewModel.LoginAsync` calls `IPersistedSessionStore.Save(...)` alongside `ISessionService.SignIn(...)` on success. `App.xaml.cs.CreateWindow()` is the actual gate: it calls `IPersistedSessionStore.TryRestore(MaxSessionAge)` first — if a non-expired session exists, it calls `ISessionService.SignIn(...)` directly and returns a `Window(AppShell)`, skipping `LoginPage` entirely; otherwise (no session, or expired — `TryRestore` self-clears an expired entry) it falls through to the original `Window(LoginPage)` path. This lives in the MAUI head (not Infrastructure) purely because `Preferences` is a MAUI Essentials API — same reasoning as WhatsApp/local notifications.

### Fase 4 — UI layer (`Pages/`, `Services/`, `Converters/`, `AppShell.xaml`, `App.xaml.cs`)

**Login → Shell swap**: the app does *not* start on `AppShell`. `App.xaml.cs` resolves `LoginPage` (DI) as the window's initial page (no Shell chrome). On successful login, `LoginPage.xaml.cs` resolves a fresh `AppShell` via `IServiceProvider` and replaces it: `Application.Current!.Windows[0].Page = shell;`. There is no session/logout flow yet — that's for whenever real multi-user auth arrives.

**Tab bar icons** (`Resources/Images/icon_*.svg`): each `ShellContent`'s `Icon` attribute points to a small (24×24) single-color line-art SVG (`icon_agenda`/`icon_clients`/`icon_catalog`/`icon_payments`/`icon_reports`/`icon_settings`, drawn in the brand `Primary` blue `#2563EB`) processed through the same `MauiImage Include="Resources\Images\*"` glob already used for `dotnet_bot.png` — no icon-font/library dependency added, matching the project's existing no-new-dependencies preference (see the custom chart controls note below for the same reasoning applied elsewhere). `icon_catalog.svg` is a scissors glyph rather than a generic list/grid icon, since this tab is specifically a beauty-salon service catalog.

**`AppShell.xaml`** is a `TabBar` with 6 tabs (`agenda`/`clients`/`catalog`/`paymentmethods`/`reports`/`settings`, the last two added in Fase 5), `FlyoutBehavior="Disabled"`. Agenda is the first tab = the default/home screen, per the brief ("weekly agenda, not a dashboard"). Non-tab pages (forms, detail, modals) are registered as plain routes in `AppShell.xaml.cs`'s constructor via `Routing.RegisterRoute("route/name", typeof(SomePage))` — add new ones there, not as `ShellContent`.

**DI-resolved page navigation**: every `Page` and `ViewModel` is registered `AddTransient` in `MauiProgram.cs`. Shell/`{DataTemplate pages:SomePage}`/`Routing.RegisterRoute` all resolve page instances through the app's `IServiceProvider`, so a page's constructor can take its ViewModel (and anything else) as a normal DI dependency — no manual `new SomePage(new SomeViewModel(...))` anywhere. When adding a new page: register both the page and its ViewModel in `MauiProgram.cs`, or navigation will throw at resolve time.

**Passing data on navigation**: pages that need an id/object use `[QueryProperty(nameof(X), "X")]` + a public settable property `X` that forwards into the ViewModel (e.g. `ReschedulePage.AppointmentId`, `CategoryFormPage.Category`). The Shell `GoToAsync(route, new Dictionary<string, object> { ["X"] = value })` overload passes the object by reference in-process — you can pass whole DTOs (not just primitive ids), which `CategoryFormPage`/`ServiceFormPage`/`PaymentMethodFormPage` rely on to prefill their edit forms instead of re-fetching.

**Appointment actions live in the agenda, not a detail page**: tapping an appointment in `AgendaPage` opens a `DisplayActionSheetAsync` whose options are computed from the appointment's current `AppointmentStatus` (`AgendaPage.BuildAvailableActions`) — Confirm/Start/Cancel/NoShow call the ViewModel directly; Finish/Reschedule navigate to their dedicated pages (they need more input than a confirm dialog). Extend `BuildAvailableActions` rather than adding a separate "appointment detail" page.

**Known Shell navigation gotcha already hit**: routes registered via `Routing.RegisterRoute` only support *relative* navigation (`Shell.Current.GoToAsync("route")`, no prefix) — per Microsoft's own Shell navigation docs, absolute routing (`//route`/`///route`) "doesn't work with pages that are registered with the `Routing.RegisterRoute` method" and throws `Global routes currently cannot be the only page on the stack` if attempted. The *actual* bug this project hit was a route-naming collision: `Routing.RegisterRoute("clients/form", ...)` shared its first path segment with the real `clients` tab route, so MAUI's relative-route resolver misidentified it as a shell-hierarchy path and threw `Relative routing to shell elements is currently not supported`. The fix was renaming the colliding registered routes so no registered route's first segment matches any `TabBar` route name — `clients/detail`→`client-detail`, `clients/form`→`client-form`, `catalog/category-form`→`category-form`, `catalog/service-form`→`service-form`, `paymentmethods/form`→`paymentmethod-form` (see `AppShell.xaml.cs`) — then every `GoToAsync` call site uses the plain relative route with no prefix at all. `appointment/*` routes never collided (no `appointment` tab exists) and were never broken. When adding a new `Routing.RegisterRoute`, pick a name that doesn't start with any existing `TabBar` route's name.

**Known XAML/MVVM gotcha already hit once**: `IsVisible="{Binding SomeObservableCollection, Converter=...}"` never refreshes after the initial bind — the collection *reference* never changes, so no `PropertyChanged` fires when you `Add`/`Clear` it. Bind to `SomeObservableCollection.Count` instead (`ObservableCollection<T>` does raise `PropertyChanged("Count")` on mutation) with `Converters/CountToBoolConverter`. `ItemsSource` bindings on `CollectionView` don't have this problem (`CollectionView` subscribes to `CollectionChanged` directly) — this only bites plain scalar bindings like `IsVisible`/`Text`.

**Display-only helpers stay in `ViewModels/`, not `Domain`**: `AppointmentStatusDisplay`/`ScheduleBlockTypeDisplay` map the English enums to Spanish labels (and `AppointmentStatusDisplay` also maps to an accent `Color`) purely for UI — the enums themselves stay English per the project-wide rule. `AgendaEntry`/`DayAgendaGroup`/`CategoryServiceGroup` are Presentation-only wrapper types (not Application DTOs) that exist solely to give `CollectionView` a uniform/grouped shape to bind against.

**Converters** (`Converters/`, registered as keyed resources in `App.xaml`): `FavoriteColorConverter` (bool→gold/gray star), `HexToColorConverter` (`"#RRGGBB"` string→`Color`, used for category color swatches), `CountToBoolConverter` (see gotcha above). Plus CommunityToolkit.Maui's `InvertedBoolConverter`, `IsStringNotNullOrEmptyConverter`, `IsListNotNullOrEmptyConverter`, `IsNotNullConverter`.

**Deliberately deferred, not forgotten**: `IAppointmentAppService` has no `GetByIdAsync` — `FinishAppointmentPage` doesn't prefill the suggested price because of this; if that's annoying in practice, add the method rather than fetching the whole week to find one appointment. `Client.DateOfBirth` is optional in the domain but `DatePicker.Date` isn't nullable — `ClientFormViewModel` uses a `HasDateOfBirth` bool + a real `DateTime` field instead of a nullable date, gated by a checkbox in the form.

### Design system (`Resources/Styles/Colors.xaml`, `Resources/Styles/Styles.xaml`)

A "Professional Business Tool" identity (deep blue `#2563EB` primary, neutral grays, no gradients/decorative color) replaced the default MAUI purple template. Every screen goes through these tokens instead of inline colors/sizes — when adding new UI, reuse these rather than hardcoding:

- **Typography** (keyed `Label` styles): `Display`/`H1`/`H2`/`H3`/`BodyLarge`/`BodySmall`/`Caption`/`FieldCaption` (the last is for form-field captions — named to avoid colliding with the `Label` control's own implicit style key). `Headline`/`SubHeadline` still exist (now `BasedOn` `H1`/`BodyLarge`) only for backward compatibility — prefer the new names.
- **Buttons**: the default (unkeyed) `Button` style *is* the primary filled-blue 56dp button now — don't set an explicit `Style` for primary actions. `SecondaryButton` (soft-blue bg, blue text), `TertiaryButton` (text-only), `Fab` (56dp circular, for the agenda/list "+" buttons) are the other three.
- **`Border` styles**: `Card` (surface bg, 16dp radius, 16dp padding, subtle shadow — wraps every list row and dashboard stat block) and `InputContainer`/`InputContainerError` (56dp-tall boxed look wrapping bare `Entry`/`Editor`/`Picker`/`DatePicker`/`TimePicker` — MAUI's own input controls have no visible border, so every boxed-looking input in this app is actually a plain input nested inside one of these `Border`s, not a native bordered control). `Chip`/`Badge` exist for pill-shaped tags but are lightly used so far (the agenda's status badge on each card).
- **Spacing/radius tokens**: `Space4`…`Space64` (x:Double, 8pt grid) and `RadiusButton`/`RadiusCard`/`RadiusInput` (ints) / `RadiusCardShape`/`RadiusInputShape`/`RadiusBadgeShape`/`RadiusChipShape` (`CornerRadius` structs, for `Border.StrokeShape`).
- **Dark mode**: every token has a `*Dark` counterpart wired through the existing `{AppThemeBinding Light=..., Dark=...}` pattern already used throughout `Styles.xaml` — the light-mode values came from the user's brand brief; the dark-mode values were *derived* by Claude (one-shade-lighter accents, inverted neutrals) since the brief only specified light.
- **Appointment status colors** (`Colors.xaml`'s `Status*`/`Status*Container`/`Status*OnContainer` tokens, one status × {accent, badge background, badge text} × {light, dark} = 48 colors): sourced from a full UI/UX design-system spec the user provided for review, adopted because it's cheap and materially improves both light/dark contrast and accessibility over the app's original flat "same color both themes" status palette. The agenda's status badge now renders as a soft container background + on-container-colored text (not solid-fill + white text). `AppointmentStatusDisplay.ToAccentColor()`/`ToBadgeBackgroundColor()`/`ToBadgeTextColor()` mirror these tokens as hardcoded hex switches (rather than an `Application.Current.Resources` lookup, consistent with this class's existing style) and pick the light/dark branch via `Application.Current?.RequestedTheme == AppTheme.Dark` — a real theme check, not a "no BindingContext" workaround, since `AgendaEntry.AccentColor`/`BadgeBackgroundColor`/`BadgeTextColor` are plain C# `Color` values computed once per `AgendaEntry` (not `AppThemeBinding`-capable), so a live OS theme flip while the Agenda is already loaded won't repaint existing cards until the next week reload (pull-to-refresh/week nav) — an accepted edge case, not a bug to chase. Any other XAML reference to a `Status*` token (e.g. `ReportsPage`'s no-show count) must go through `{AppThemeBinding Light=..., Dark=...}` explicitly — a bare `{StaticResource StatusX}` only ever resolves the light value.
- **Font**: styles reference `InterRegular`/`InterSemibold`, but the actual Inter `.ttf` files could not be downloaded in this dev environment (network egress to the font's GitHub release/CDN hosts was blocked) — `MauiProgram.cs`'s `ConfigureFonts` currently aliases those two names to the already-bundled `OpenSans-Regular.ttf`/`OpenSans-Semibold.ttf` as a visual stand-in. To swap in the real typeface later, drop `Inter-Regular.ttf`/`Inter-SemiBold.ttf` into `Resources/Fonts/` and repoint those two `AddFont` calls — no XAML changes needed anywhere else, since every screen references the style keys, never raw font family names.
- **Custom chart controls** (`Controls/Charts/`): `DonutChartView` and `HourHistogramView` are small `GraphicsView`-based controls (bindable `ItemsSource` of the relevant Report DTO list, computing proportions/max internally in their `IDrawable.Draw`) used on `ReportsPage` for the payment-method breakdown and busiest-hours histogram — added instead of a charting NuGet package, per an explicit no-new-dependencies preference for this solo-maintained app.

### Fase 5 — Reports, Settings, i18n, local notifications, WhatsApp reminders

**Settings** (`Features/Settings/` in Application, `SettingsViewModel`/`SettingsPage`): one combined save covers three concerns in a single `AppSettings` row — language/theme/currency/date-format/time-format, the 7-row `WorkingHours` table, and the `NotificationRule` list. `SettingsViewModel` aliases `using AppTheme = BeautySalon.Domain.Enums.AppTheme; using Currency = ...Currency;` to disambiguate from `Microsoft.Maui.ApplicationModel.AppTheme` — needed anywhere a ViewModel touches both. `WorkingDayEditItem`/`NotificationRuleEditItem` are Presentation-only edit-row wrappers (not DTOs) so the `CollectionView` rows can bind `TimePicker.Time` (a `TimeSpan`) without converting back and forth against the domain's `TimeOnly`. `SettingsAppService.UpdateNotificationRulesAsync` upserts by `MinutesBefore` and never deletes rows not present in the input — fine today since the UI only ever manages the same fixed set of rules (15/30/60/1440 min), but don't assume the rule count is capped elsewhere without checking `LocalNotificationScheduler`'s `MaxRulesPerAppointment` (see below).

**Reports** (`Features/Reports/`, `ReportAppService`, `ReportsViewModel`/`ReportsPage`): one consolidated `GetSummaryAsync(from, to, professionalId)` computing revenue, completed/cancelled/no-show counts, new-vs-inactive clients (`InactivityThresholdDays = 60`), revenue-by-payment-method, and top-5 clients/services/categories/hours — all in one query rather than one AppService method per stat. Required adding `.Include(a => a.PaymentMethod)` to `AppointmentRepository.GetByDateRangeAsync` to support the payment-method grouping (it wasn't needed before Reports existed). Every `CollectionView` on `ReportsPage`/`SettingsPage` needs an explicit `HeightRequest` — nesting a `CollectionView` inside a `ScrollView`/`VerticalStackLayout` without one renders it at zero height (a MAUI layout quirk, not a bug in this app); don't remove those `HeightRequest`s when tweaking the pages.

**i18n** (`Resources/Strings/AppResources.{resx,en.resx}`): the default (unsuffixed) `.resx` is Spanish; `.en.resx` is the English satellite, picked automatically by `ResourceManager` from `CultureInfo.CurrentUICulture` — no manual switch statement anywhere. The default `.resx` uses MSBuild's `StronglyTypedFileName`/`Namespace`/`ClassName` metadata (see `Beauty Salon.csproj`) instead of the VS-only `PublicResXFileCodeGenerator` custom tool, so `dotnet build` on the CLI regenerates `AppResources.Designer.cs` correctly without Visual Studio. XAML uses `{x:Static strings:AppResources.KeyName}` (needs `xmlns:strings="clr-namespace:Beauty_Salon.Resources.Strings"` on the root element) — every page's `Title`/`Text`/`Placeholder`/`ToolbarItem.Text` should go through this, including tab titles in `AppShell.xaml`. C# code (ViewModels, code-behind) references `AppResources.KeyName` directly via `using Beauty_Salon.Resources.Strings;` — this applies to validation messages set on `ErrorMessage` too (e.g. `ViewModelBase`'s unexpected-error fallback, the "select a client/service/category/payment method" guards), not just XAML; when adding a new user-facing string anywhere, add the resx key first rather than hardcoding it, in either language. **`x:Static` bindings resolve once at page construction**, so a language change only takes effect for pages built afterward — `MauiProgram.CreateMauiApp()` applies the saved `AppSettings.Language` to `CultureInfo.Current(UI)Culture`/`DefaultThreadCurrent(UI)Culture` once at startup (`ApplySavedCulture`, right after `InitializeDatabaseAsync`), so a language switch in Settings requires an app restart to fully apply — this is a known, accepted limitation, not a bug to silently fix by adding runtime re-binding infrastructure.

**Local notifications** (`Services/IAppointmentNotificationScheduler` + `LocalNotificationScheduler`, using `Plugin.LocalNotification`): lives in the MAUI head, not Infrastructure, since it depends on the plugin's Android/iOS-specific types. The real types are namespaced `Plugin.LocalNotification.Core.Models.*` / `...Core.Models.AndroidOption.*` (not the plugin's root namespace) — reflecting the package's own XML doc comments was the only reliable way to confirm this; don't assume flat `Plugin.LocalNotification.NotificationRequest` from memory. `ScheduleRemindersAsync` reads the enabled `NotificationRule`s from `ISettingsAppService`, schedules one `NotificationRequest` per enabled rule (skipping any whose computed fire time has already passed), and derives each notification's `NotificationId` deterministically from the appointment's `Guid` + a slot index (0–7, `MaxRulesPerAppointment`) via `GetNotificationId` — this means `CancelRemindersAsync` can cancel by recomputing the same ids without needing a lookup table, but it also means at most 8 reminders per appointment are supported. Wired at every lifecycle point that changes an appointment's timing/validity: create and reschedule schedule (reschedule also cancels the old appointment's reminders, since reschedule creates a *new* `Appointment` row — see the note on `RescheduleAsync` above); cancel/no-show/finish cancel. **Important**: `AppointmentFormViewModel`/`RescheduleViewModel`/`FinishAppointmentViewModel` call `IAppointmentAppService` directly (not through `AgendaViewModel`'s `CreateAppointmentAsync`/`RescheduleAppointmentAsync`/`FinishAppointmentAsync` wrapper methods, which exist but are unused/dead code) — the notification-scheduler calls were added at the real call sites, not on the unused `AgendaViewModel` wrappers; don't assume wiring something into those wrapper methods has any effect. The Android notification channel (`LocalNotificationScheduler.AppointmentReminderChannelId`) is created once via `.UseLocalNotification(config => ...)` in `MauiProgram.cs`; the `POST_NOTIFICATIONS` runtime permission (Android 13+) is requested from `AgendaPage.OnAppearing` since Agenda is the home screen users land on right after login.

**Brand assets** (`Resources/AppIcon/*.svg`, `Resources/Splash/splash.svg`, `Platforms/Android/Resources/drawable/ic_notification.xml`): the app icon/splash no longer use the default MAUI purple template mark — both now share a simple calendar+checkmark glyph (white shape, blue `#2563EB` checkmark) matching the design system's brand color, with `Beauty Salon.csproj`'s `MauiIcon`/`MauiSplashScreen` `Color` attributes updated to match. `splash.svg` is kept byte-identical in content to `appiconfg.svg` by convention (update both together). The Android notification status-bar icon (`ic_notification.xml`, a hand-written vector drawable — Android forces these to render as a flat single-tone silhouette regardless of any color set here) uses the same calendar+check motif and is wired via `LocalNotificationScheduler`'s `Android.IconSmallName = new AndroidIcon("ic_notification")`; without it, reminders fall back to the plugin's own generic icon. `MainPage.xaml`/`.xaml.cs` and `Resources/Images/dotnet_bot.png` are unused default-template leftovers (the app starts on `LoginPage`, never `MainPage`) — harmless but safe to delete if cleaning up.

**Data backup/restore** (`Services/IDataBackupService`/`DataBackupService`, `Services/DatabasePaths`): "Backup"/"Restore" buttons on `SettingsPage` (below Save). Backup copies the live `beautysalon.db3` to `FileSystem.CacheDirectory` with a timestamped name and hands it to the platform share sheet (`Share.Default.RequestAsync`) — no custom export format, the raw SQLite file *is* the backup, so fidelity is exact and there's no serialization logic to maintain. Restore uses `FilePicker.Default.PickAsync` (no file-type filter — cross-platform MIME filtering for a non-standard `.db3` extension is unreliable, so any file can be picked and `DataBackupService` validates it's a real Beauty Salon backup itself, via a raw `Microsoft.Data.Sqlite.SqliteConnection` checking for a `Users` table) before overwriting the live file; both directions run `PRAGMA wal_checkpoint(TRUNCATE)` first so a plain file copy captures a complete, consistent snapshot instead of missing recent writes still sitting in the `-wal` sidecar. Restore is destructive and irreversible by design (single-user app, no undo) — `SettingsPage.xaml.cs`'s `OnRestoreClicked` shows a confirmation `DisplayAlertAsync` before calling `RestoreAsync`, and the success message tells the user to restart the app (matching the same restart-required pattern already used for language changes) rather than trying to hot-swap the live EF Core connection.

**In-app manual (`Pages/HelpPage.xaml`)**: a reference/cheat-sheet page reachable via a "Ayuda"/"Help" `ToolbarItem` on `SettingsPage`, registered as route `"help"`. Not a step-by-step UI walkthrough — the app has exactly one user (the salon owner), who already knows the screens; the page exists so *they* can look up a non-obvious business rule they've forgotten (e.g. "why did rescheduling create a second appointment"). Content is 7 collapsible `CommunityToolkit.Maui.Views.Expander` sections (one per tab, in tab order, plus a "Preguntas frecuentes"/FAQ section), each wrapped in the design system's `Card` Border style. `HelpPage` deliberately has **no ViewModel** — every string is static `x:Static strings:AppResources.X`, and each section's body is a single resx key containing all its bullets (not one key per bullet) to keep the translation surface small. FAQ entries are drawn from real confusions hit while testing this app, not speculative ones — extend it the same way (only add an entry once something has actually confused the owner in practice).

**WhatsApp reminder button**: `AgendaPage.BuildAvailableActions` adds a `SendReminderAction` option (only when the appointment's client has a phone number) alongside Confirm/Start/Reschedule/etc.; picking it builds a `wa.me/<digits>?text=<url-encoded message>` link via `AppResources.WhatsAppReminderMessageFormat` and opens it with `Launcher.Default.OpenAsync`. This is separate from `ClientDetailViewModel.SendWhatsAppAsync` (a generic contact-the-client button with no pre-filled text) — the agenda one is appointment-specific and reminder-worded on purpose; don't merge them.

### Fase 6 — Testing (`BeautySalon.Application.Tests/`)

One xUnit test project so far, covering the two most business-rule-heavy AppServices plus the `Rut` value object. Not integration/UI tests, not a coverage-completeness effort — started with the highest-value, most-likely-to-regress logic first (schedule overlap/working-hours rules, appointment status transitions, RUT check-digit validation, client duplicate-RUT detection).

**Real SQLite, not mocks, not EF Core's InMemory provider** (`TestDatabase.cs`): each test class owns a fresh `TestDatabase` (`IDisposable`) backed by an open `SqliteConnection("Data Source=:memory:")` (kept open for the connection's lifetime — an in-memory SQLite DB is deleted the moment its last connection closes) + `Context.Database.EnsureCreated()`, with all 10 concrete repositories + a real `UnitOfWork` constructed directly (no DI container involved in tests). This is deliberate, not incidental: the repo already documents one real bug (`ClientRepository.SearchAsync`'s RUT-conversion SQL-translation failure) that only surfaces against a real relational provider — EF Core's InMemory provider or a fully-mocked `IUnitOfWork` would both have masked it. The audit-stamping interceptor is intentionally **not** wired into test contexts (adds no value to the tests written so far); add it only if a future test needs to assert on Created/UpdatedBy stamps.

**A real bug this suite caught on its first run**: `WorkingHours.ProfessionalId` and `Appointment.ProfessionalId` both have a genuine FK to `User.Id`. An early draft of `AppointmentAppServiceTests` used a bare `Guid.NewGuid()` for the test professional with no backing `User` row, and every test failed with `SQLite Error 19: FOREIGN KEY constraint failed` — fixed by seeding a real `User` row with `Id = _professionalId` in `SeedBasicDataAsync`. Keep seeding a real `User` for any new professional-scoped test; a loose `Guid` will not work against this schema.

**Running**: `dotnet test "BeautySalon.Application.Tests/BeautySalon.Application.Tests.csproj"`. The project is excluded from `Beauty Salon.csproj`'s compile glob the same way the other four sibling projects are (`<Compile Remove="BeautySalon.Application.Tests\**\*.cs" />`) — anyone adding a *sixth* sibling class library needs the same exclusion line, per the note already in that `ItemGroup`.

**Not covered yet**: `ClientAppService`/`AppointmentAppService` are the only two AppServices under test; `CatalogAppService`, `PaymentMethodAppService`, `SettingsAppService`, `ReportAppService`, `AuthAppService` have none. No ViewModel-level tests (would need a fake `ICurrentUserContext`/`IAppointmentNotificationScheduler` etc.), no UI/Shell-navigation tests, no Persistence-layer-specific tests beyond what the AppService tests exercise incidentally.
